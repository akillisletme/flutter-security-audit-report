## КАК ТЫ БУДЕШЬ ИСПОЛЬЗОВАТЬ ЭТОТ ФАЙЛ (команда для Claude Code)

Этот файл — это **инструкция по аудиту**, а не жёсткий чек-лист. В разных проектах используются разные технологии (в одном проекте — Supabase, в другом — только Firebase, в третьем — обе сразу). Скажи Claude Code, запущенному в корневом каталоге приложения, следующее:

> «В корневом каталоге проекта есть `GUVENLIK_DENETIM_RAPORU.md`. Сначала изучи проект:

> 2. Просканируй структуру каталогов `android/` и `ios/`, файлы manifest/plist.
> 3. Затем последовательно примени все контрольные пункты из этого отчёта. Для каждого пункта прочитай соответствующие файлы, ищи паттерны красных флагов. Пункты, которым нет соответствия в проекте, помечай как «НЕ ПРИМЕНИМО», не выдумывай.
> 4. Каждую найденную уязвимость записывай в файл `GUVENLIK_BULGULARI.md` согласно **Формату вывода** в самом конце.
> **НЕ ИЗМЕНЯЙ код. Только выявляй и составляй отчёт.** Исправления будем делать по мере того, как я буду одобрять их пункт за пунктом.»

Claude Code должен сначала провести выявление, прежде чем писать код. Сначала увидим все уязвимости, а исправления сделаем по мере одобрения пользователем. Каждый пункт, в котором ты не уверен (настройки вне кода, например в Cloud/дашборде), помещай в список **«РУЧНАЯ ПРОВЕРКА»**.

**Для глубокого сканирования (чтобы LLM не проходил поверхностно/лениво):** При изучении каждого файла или модуля не прыгай сразу к выводу. Сначала открой тег `<анализ>` и рассуждай пошагово: *что я ищу в этом файле, что я вижу в коде, какому правилу отчёта (A1 / C3 / ...) соответствует эта ситуация, совпал ли какой-либо паттерн красного флага?* По окончании анализа запиши находку в `GUVENLIK_BULGULARI.md`. Кроме того, **не пропускай ни один пункт молча:** в конце отчёта сформируй **список самопроверки (покрытия)**, в котором каждый пункт от A1 до F4 отмечен как «оценён / НЕ ПРИМЕНИМО / РУЧНАЯ ПРОВЕРКА» — так докажи, что не осталось неоценённых пунктов. В большой кодовой базе продвигайся раздел за разделом (по завершении каждого крупного раздела давай краткое промежуточное резюме), не проходи всё поверхностно за один раз.

# ПЕРСПЕКТИВА АТАКУЮЩЕГО — РЕАЛЬНОСТЬ КЛИЕНТСКИХ ЗАЩИТ В 2026 ГОДУ

Этот раздел — не контрольный пункт, а **модель угроз.** Оценивая пункты ниже, Claude Code (и разработчик) должны знать следующую реальность: по состоянию на 2026 год инструменты в руках упорного атакующего против Flutter-приложения зрелы и в значительной мере автоматизированы. Готовый инструментарий, созданный исследовательскими сообществами на разных языках (английский, русский, китайский, испанский, немецкий), делает следующее:

**Статический анализ / извлечение кода:**
- **reFlutter** (официальный инструмент в OWASP MASTG): патчит `libapp.so`, выгружает информацию о классах/функциях/именах Dart, обходит некоторые реализации cert-pinning в Flutter **без необходимости root**. (Примечание: в некоторых версиях Flutter встроенный proxy IP был удалён, теперь требуется настройка proxy на устройстве — то есть это не блокировка, а лишь ещё один шаг. Поведение, специфичное для версии, подтверждай в момент аудита.)
- **Blutter**: разбирает метаданные Dart AOT из `libapp.so`; генерирует имена символов для IDA/Ghidra, дамп object pool и готовый шаблон Frida. В приложении без обфускации имена классов/методов становятся читаемыми.
- **Ghidra/IDA + скрипт Blutter**: криптографические детали, такие как режим AES, использование SHA-256 и даже встроенный приватный ключ RSA, можно извлечь из дизассемблирования.

**Динамический анализ / рантайм:**
- **Frida + objection**: одной командой (`ios sslpinning disable`, `android sslpinning disable`) — обход pinning, **дамп keychain**, обход файловой системы, обход биометрической проверки. **Без необходимости jailbreak/root** с помощью `patchipa`/`patchapk` можно встроить frida-gadget в приложение. (Версии инструментов подтверждай на актуальном релизе в момент аудита; здесь речь идёт не о конкретной версии, а о возможности.)
- **Многоязычные репозитории готовых скриптов**: «универсальные» скрипты, обходящие одновременно детекцию root + SSL pinning + эмулятор + отладку, доступны публично (особенно распространены в китайских и английских сообществах). Детекция root/jailbreak сама по себе почти никогда не достаточна — её можно обойти.
- **Даже библиотеки jailbreak-детекции, такие как IOSSecuritySuite / freerasp,** обходятся через Frida-хук или пропатченный бинарник (недавние работы команд вроде CyberCX и Appknox это показывают).

**Результат из реального мира (опубликованный кейс):** Находящееся в продакшене Flutter-приложение (Android) было поэтапно полностью подвергнуто обратной разработке — **был извлечён встроенный приватный ключ RSA, восстановлен кастомный формат подписи RSA-SHA256, и продакшен-API был успешно вызван из независимого Python-клиента.** То есть предположение приложения «я отклоняю запрос, пришедший не от моего клиента» рухнуло.

### Три правила защиты, вытекающие отсюда (основа, на которой держится весь отчёт)

1. **Всё, что находится на клиенте, считается читаемым.** Секрет, ключ, логика подписи, бизнес-правило — всё это извлекаемо. Поэтому пункты A1 (секреты) и A11 (крипто) нельзя закрыть фразой «я обфусцировал, значит в безопасности».
2. **Клиентские защиты (обфускация, pinning, детекция root/JB) не *предотвращают* атаку, а делают её *дороже*.** Они не бесполезны — отсеивают массовые/автоматизированные атаки, повышают стоимость атаки — но как единственная линия обороны никогда не достаточны. Они должны быть слоями (defense-in-depth).
3. **Единственная настоящая граница безопасности — это backend.** Авторизация, владение данными, проверка оплаты/премиума, ограничение частоты запросов должны применяться **на сервере** (Firestore Rules / RLS / Cloud Function + App Check/Play Integrity). Атакующий вызывает не твоё приложение, а напрямую твой API — API должен уметь проверять этот вызов самостоятельно.

> **Claude Code должен использовать это так:** Для каждой находки указывай, относится ли контроль к «клиентской защите» или к «серверному принуждению». Если клиентская защита в одиночку охраняет критическое решение (премиум, права доступа, оплата, доступ к данным), а на backend соответствия нет → чётко напиши это в разделе «Влияние» находки как **«обходится на стороне клиента, серверная проверка отсутствует»**.

# РАЗДЕЛ A — СТОРОНА КЛИЕНТА (FLUTTER / DART)

Фундаментальная истина такова: **скомпилированный APK/IPA — это публично доступный файл.** С помощью инструментов вроде `jadx`, `apktool`, `reFlutter`, `blutter` файл `libapp.so` декомпилируется; компиляция Dart AOT это усложняет, но не предотвращает. Ничего, что находится на клиенте, не считай «секретным». В OWASP Mobile Top 10 2024 эту реальность подчёркивают **M7 (Insufficient Binary Protections)** и **M8 (Security Misconfiguration)**.

> **Карта категорий OWASP 2024 (правильные номера):** M1 Improper Credential Usage · M2 Inadequate Supply Chain Security · M3 Insecure Authentication/Authorization · M4 Insufficient Input/Output Validation · M5 Insecure Communication · M6 Inadequate Privacy Controls · M7 Insufficient Binary Protections · M8 Security Misconfiguration · M9 Insecure Data Storage · M10 Insufficient Cryptography

## A1 — 🔴 Встроенные секреты / API-ключи (OWASP M1: Improper Credential Usage)

**Номер 1** в списке OWASP 2024 и всё ещё первый в 2026 году. Самая частая и самая дорогая ошибка. Утечка Uber 2017 года (ключ S3 в репозитории), утечка API-ключа Strava, hardcoded-учётные данные, найденные в десятках популярных Android/iOS-приложений в 2024 году, — всё из этой категории.

**Claude Code должен просканировать следующее:**
- Все `.dart`-файлы в `lib/`, `assets/`, `.env`, `pubspec.yaml`
- `android/app/google-services.json`, `ios/Runner/GoogleService-Info.plist`
- `android/local.properties`, `android/app/build.gradle(.kts)`, `ios/Runner/Info.plist`
- `firebase_options.dart` (сгенерированный FlutterFire CLI)
- История Git: с помощью `git log -p | grep -iE '(apikey|secret|token|password|service_role)'` — ключи, которые впоследствии удалены, но остались в истории

**Паттерны красных флагов (regex/grep):**
```
apiKey|api_key|apikey|secret|password|passwd|token|bearer|private_key|client_secret
AIza[0-9A-Za-z_\-]{35}        → Google/Firebase API key
sk_live_|sk_test_|pk_live_    → Stripe key
eyJ[A-Za-z0-9_\-]+\.eyJ       → JWT (критический, если содержит service_role)
service_role                  → сервисный ключ Supabase НИКОГДА не должен быть на клиенте
supabaseKey|SUPABASE_SERVICE|SUPABASE_URL
ghp_|gho_|github_pat_         → GitHub token
xox[baprs]-                   → Slack token
-----BEGIN (RSA|EC|OPENSSH) PRIVATE KEY-----
```

**Важные различия (не создавай ложных срабатываний):**
- **Firebase `apiKey`** (в google-services.json / firebase_options.dart): технически «не секрет», он идентифицирует проект. Официальная документация Firebase гласит: ограниченные ключи к сервисам Firebase не считаются секретом. **НО если он не ограничен** (см. A2) или открыт для перечисления пользователей (user enumeration), им злоупотребляют. Помечай находку как «требуется проверка ограничений», а не «утёкший секрет».
- **Supabase `anon` key**: намеренно публичный, сам по себе не проблема — **НО если RLS отключён, это катастрофа** (см. раздел C). Привязывай находку к состоянию RLS.
- **Supabase `service_role` key**: если находится на клиенте, **сразу 🔴 КРИТИЧЕСКИЙ** — он обходит весь RLS, даёт полный доступ к базе данных.
- **API-ключ Google Maps / Places / Gemini**: его наличие в `AndroidManifest.xml`/`Info.plist`/коде иногда обязательно, но **если он не ограничен именем пакета + SHA-1 (Android) или bundle ID (iOS)**, кто-то другой израсходует твою квоту → счёт оплатишь ты. Он должен быть **отдельным** от ключа Firebase и ограничен соответствующим API.

**Правильный подход:** Настоящие секреты хранятся на backend (Cloud Functions / Supabase Edge Functions); клиент запрашивает временный токен. `flutter_dotenv`, `--dart-define`, `--dart-define-from-file` предназначены только для **нечувствительной** конфигурации — они читаемы из скомпилированного бинарника, это не средство безопасности.

## A2 — 🟠 Отсутствие ограничений API-ключа (вне кода, РУЧНАЯ ПРОВЕРКА)

**Проверка:** В Google Cloud Console для каждого ключа:
- Есть ли ограничение приложения? (Android: имя пакета + подпись SHA-1; iOS: bundle ID; Web: HTTP referrer)
- Есть ли ограничение API? (Разрешены только используемые API)
- Отделён ли ключ Firebase от ключа Maps/Gemini?

Claude Code не может увидеть настройки Cloud Console, читая код — этот пункт нужно пометить как **«РУЧНАЯ ПРОВЕРКА»** и напомнить о нём пользователю. Неограниченный ключ = злоупотребление квотой, спам-аккаунты, перечисление данных, неожиданный счёт.

## A3 — 🔴 Небезопасное локальное хранилище (OWASP M9: Insecure Data Storage)

Критическая истина: **Hive (hive/hive_ce), SharedPreferences, SQLite по умолчанию не зашифрованы** — это открытый текст. На устройстве с root/jailbreak или через ADB backup читаются из `/data/data/<пакет>/`. Исследование Talsec: даже JWT Firebase Auth хранится в **открытом тексте** в `shared_prefs/com.google.firebase.auth.*.xml` и может быть украден на устройстве с root и использован для impersonation.

**Claude Code должен просканировать следующее:**
- Все места, где пишут `SharedPreferences`, `Hive`/`hive_ce`, `sqflite`, `isar`
- **Что именно пишется** в эти хранилища: токен, пароль, email, телефон, ИНН/удостоверение, геолокация, сессия, карта/IBAN → чувствительно
- Есть ли в проекте **`flutter_secure_storage`?** Если нет, а чувствительные данные пишутся локально → находка.

**Красный флаг:**
```
prefs.setString(...token...) / setString(...password...) / setString(...jwt...)
Поля Hive box / SharedPreferences: authToken, refreshToken, accessToken,
    session, jwt, credit, card, iban, phone, email, tckn, otp, pin, apiKey
запись token/PII в файл через writeAsString(...) (в каталоги path_provider)
```

**Правила:**
- Чувствительные данные (токен, сессия, идентификация, оплата) → **`flutter_secure_storage`** (iOS Keychain / Android Keystore + EncryptedSharedPreferences).
- Если чувствительные данные должны оставаться в Hive → **зашифрованный AES box** (`HiveAesCipher`), а ключ шифрования хранится в **`flutter_secure_storage`** (не встраивается в код).
- **Нечувствительные** офлайн-данные вроде счёта/кубика/темы/языка/последней вкладки → простой Hive/SharedPreferences **в порядке**.
- Примечание: хранение токена Firebase SDK в открытом тексте — это поведение SDK; защита от этого — **целостность устройства** (App Check / Play Integrity) + **короткое время жизни токена + отзыв токена при logout** (см. B и C).
- **Кеш офлайн-персистентности Firestore тоже не зашифрован** (`Settings(persistenceEnabled: true)` — во Flutter включён по умолчанию). Каждое данное Firestore, которое читает приложение, накапливается на устройстве в открытом файле. Для очень чувствительных коллекций (сообщения, оплата, идентификация) рассмотри отключение персистентности либо слой блокировки устройства/дополнительного шифрования; как минимум проверь, вызывается ли **`clearPersistence()` при logout** (если не вызывается, данные предыдущего пользователя остаются на устройстве → утечка на общем устройстве, 🟠).

Claude Code должен перечислить каждое поле, записываемое во все локальные хранилища, и классифицировать их как **чувствительное / нечувствительное**.

## A4 — 🔴/🟠 Небезопасная связь (OWASP M5) — HTTPS и certificate pinning

**Сканирование:**
- Все эндпоинты, начинающиеся с `http://` (не HTTPS) → 🔴
- `android:usesCleartextTraffic="true"` в `AndroidManifest.xml` → 🟠
- `cleartextTrafficPermitted="true"` или ослабленные `trust-anchors` в `res/xml/network_security_config.xml` → 🟠
- iOS `Info.plist`: `NSAllowsArbitraryLoads = true` под `NSAppTransportSecurity` (обход ATS) → 🟠
- Если в Dio/http client `badCertificateCallback` всегда возвращает `true` или `HttpClient` `onBadCertificate = (_,__,___) => true` → 🔴 (проверка сертификата отключена, открыто для MITM)

**Рекомендация:** TLS 1.2+ обязателен, cleartext запрещён. В потоках, обрабатывающих чувствительные данные, — **certificate pinning** (против MITM в публичном Wi-Fi). Трафик Firebase SDK уже идёт по TLS; если есть дополнительные вызовы `dio`/`http`, то pinning + добавление токена через interceptor, отсутствие утечки чувствительных данных в логах ошибок.

> **Реалистичное ожидание:** Pinning отсеивает массовый/автоматизированный MITM и новичка-атакующего, но reFlutter (патч `libapp.so`) и objection (`ios/android sslpinning disable`) готовыми командами могут обойти многие реализации pinning **без root/jailbreak**. То есть pinning — это не гарантия «мой трафик никогда не будет виден»; **сервер всё равно должен проверять каждый запрос самостоятельно** (App Check/Play Integrity + auth). Добавить pinning хорошо, а *полагаться* на него — ошибка.

## A5 — 🟡 Логирование чувствительных данных / включённый debug-режим (OWASP M8)

**Сканирование:**
- Строки, в которых внутри `print(...)`, `debugPrint(...)`, `developer.log(...)`, `logger.*` встречаются токен/пароль/PII/полное тело ответа
- Многословное логирование, работающее в продакшене без проверки `kReleaseMode`/`kDebugMode`
- Отправка PII в Crashlytics/Sentry как custom key/log (`setCustomKey`, `recordError`, `log`) — отправка email/телефона/токена
- `android/app/build.gradle`: если в release-сборке осталось `debuggable true` → 🟠
- Если Dio `LogInterceptor` остался включённым в release (логирует все запросы/ответы) → 🟠

**Правило:** В release логи выключены/замаскированы. Не отправляй сырые PII в Crashlytics.

## A6 — 🟡 Отсутствие защиты от reverse engineering / обфускации (OWASP M7: Insufficient Binary Protections)

**Сканирование:**
- Используется ли в build-скриптах / CI `flutter build ... --obfuscate --split-debug-info=<dir>`?
- Android `android/app/build.gradle`: есть ли `minifyEnabled true` + `shrinkResources true` + правила R8/ProGuard?

**Примечание:** Обфускация — не 100% защита, она лишь повышает стоимость атаки. **Главное правило: перенеси чувствительную бизнес-логику и авторизацию на backend, считай клиент недоверенным.** Безрисковая игровая/UI-логика вроде «достиг ли 101», «какой звук проиграть» может быть на клиенте; но «премиум ли этот пользователь / валидна ли эта операция / реален ли этот счёт» должно проверяться на backend. Опирай критические проверки не только на обфускацию, но и на серверную проверку.

> **Конкретно:** Если обфускации НЕТ, Blutter делает имена классов/методов из `libapp.so` читаемыми и генерирует готовый скрипт символов для IDA/Ghidra — код читается почти на уровне исходников. Если обфускация ЕСТЬ, это усложняется, но не становится невозможным (имена исчезают, логика остаётся). Поэтому `--obfuscate` **должен быть** (особенно если критическая бизнес-логика на клиенте), но сам по себе не считается достаточным.

## A7 — 🟡 Избыточные разрешения (Over-permissioning) + детекция Root/Jailbreak и Tamper

**Сканирование:** Разрешения в `AndroidManifest.xml` и `Info.plist` против того, какие функции приложение реально использует.
- Действительно ли нужен `ACCESS_FINE_LOCATION`, или достаточно `ACCESS_COARSE_LOCATION` (город/приблизительно)? (В функции карты/объявлений **приблизительная геолокация вместо точного GPS** правильна и с точки зрения приватности, и с точки зрения законодательства о персональных данных.)
- `READ_SMS`, `READ_CONTACTS`, `CAMERA`, `RECORD_AUDIO`, `MANAGE_EXTERNAL_STORAGE`, фоновая геолокация, `QUERY_ALL_PACKAGES` → если не используется, удали (магазин тоже может отклонить).
- Есть ли для каждого iOS-разрешения строка описания в `Info.plist` (`NSLocationWhenInUseUsageDescription`, `NSCameraUsageDescription` и т.д.)? Если нет — приложение падает + отклонение в магазине.
- Есть ли **детекция root/jailbreak/эмулятора** (в платёжных/чувствительных приложениях)? Play Integrity / App Attest / `freerasp` и т.п. Если нет — для чувствительных потоков 🟡.

> **Предупреждение:** Детекция root/JB *сама по себе* не является безопасностью. Её рутинно обходят через Frida/objection и через «универсальные bypass»-скрипты, гуляющие по многоязычным сообществам; даже библиотеки вроде IOSSecuritySuite/freerasp можно пропатчить. Её ценность: предупредить честного пользователя о рискованной среде + усложнить автоматизированный/массовый фрод. Критическое решение (подтверждение оплаты, перевод денег) **никогда** не должно опираться на предположение «устройство выглядело чистым»; основой должны быть серверная проверка + сигнал целостности устройства (Play Integrity/App Attest, оценивается на backend).

## A8 — 🟠 Недостаток валидации ввода (OWASP M4)

**Правило:** Клиентскую валидацию можно обойти — **каждая валидация должна повторяться на сервере.** В UGC-полях вроде текста объявления, имени пользователя, комментария, сообщения:
- Есть ли ограничение длины/формата/типа и на сервере (Firestore Rules `request.resource.data...` / Supabase policy / Edge Function)?
- Если используется WebView, безопасны ли `javascriptMode` и `onNavigationRequest`? Рендерится ли инъекция сырого HTML/URL (XSS)?
- Валидируются ли URL, открываемые через deep link / `url_launcher` (см. A10)?
- SQL/NoSQL: встраивается ли пользовательский ввод напрямую в запрос?
- **GraphQL (если в проекте есть `graphql`/`graphql_flutter`/`ferry`/`artemis`; если нет — НЕ ПРИМЕНИМО):**
  - Отключён ли **introspection на backend в продакшене**? Если включён, атакующий извне читает всю схему, имена типов и полей и составляет карту поверхности атаки → 🟠.
  - Есть ли **лимит глубины / сложности запроса (query depth / complexity)** против чрезмерно глубоких/вложенных (nested) запросов? Если нет, одним глубоким запросом можно вызвать DoS + взрыв счёта → 🟠.
  - Авторизация на уровне полей выполняется **на сервере**? Запрос приходит от клиента; то, какое поле кому возвращается, должно ограничиваться на backend (скрытие поля на клиенте — это не авторизация — см. B4).

## A9 — 🟡 Сторонние зависимости (OWASP M2: Supply Chain)

- Вывод `flutter pub outdated` / `dart pub outdated` — пакеты с известными CVE/заброшенные
- Пакеты в `pubspec.yaml`, у которых прекращена поддержка, мало загрузок, подтягиваемые через `git:`/`path:`
- Незакреплённые версии (`any`, слишком широкие диапазоны `^`/`>=`)

- **Typosquatting (поддельный/похоже названный пакет):** Есть ли среди имён пакетов в `pubspec.yaml` имя, намеренно похожее на официальные/популярные пакеты (напр. `url_launcher` → `url_launcer`, `firebase_core` → `firebase_cor`, `flutter_secure_storage` → `flutter_secure_storge`)? **Визуальный осмотр глазами ненадёжен** — сверяй каждое имя пакета с реальным пакетом на pub.dev: совпадают ли verified publisher, число загрузок/лайков, дата последнего релиза и адрес репозитория? ## A10 — 🟠 Безопасность Deep Link / URL Launcher / Share

Этот пункт для проектов, использующих `url_launcher`, `app_links`/`uni_links`, `share_plus`, WebView.

**Сканирование:**
- **Источник** URL в вызовах `launchUrl(...)`: если он приходит из пользовательского контента (ссылка объявления, ссылка профиля из Firestore/Supabase) и схема не валидируется → атакующий может внедрить `javascript:`, `file:`, `intent:`, фишинговую `https://` ссылку → 🟠. Должен быть allow-list только `https`/`mailto`/`tel`.
- Экраны, открываемые через **deep link / App Link (Android) / Universal Link (iOS)**: выполняется ли напрямую авторизованное действие с приходящим параметром? (Напр. если `myapp://reset?token=...` обрабатывается без проверки — это путь к захвату аккаунта.)
- Корректны ли `intent-filter` `android:autoVerify="true"` и `assetlinks.json` в `AndroidManifest.xml`? (Против App Link hijack.)
- Есть ли **утечка PII/токена** в контенте, шарящемся через `share_plus`? (Изображения вроде «карточки конца игры» обычно безрисковы; но в шарящемся тексте не должно быть email/телефона/deep link + токена.)
- WebView: валидация `Uri`, allow-list `onNavigationRequest`, отключены `allowFileAccess`/`allowUniversalAccessFromFileURLs`.
- **JS-мост WebView ↔ Dart:** Если через `addJavaScriptChannel(...)` / `JavascriptChannel` открывается канал отправки сообщений веб-контента на нативную/Dart-сторону, то JS, работающий внутри WebView (потенциально под контролем атакующего), может вызвать этот канал → если функция моста **выполняет чувствительную операцию/доступ к данным с невалидированным вводом** → 🟠. Обязателен доступ к мосту только с доверенных origin + строгая валидация приходящего сообщения.

## A11 — 🟠 Слабая / неправильная криптография (OWASP M10: Insufficient Cryptography)

Если в проекте есть любое шифрование/хеширование/подпись (пакеты: `crypto`, `encrypt`, `pointycastle`, `cryptography` и т.д.):

**Красные флаги:**
```
md5(...) / sha1(...)                    → для пароля/целостности 🔴 (сломанные алгоритмы)
AESMode.ecb / 'AES/ECB'                 → 🔴 (протекают паттерны; используй GCM/CBC+HMAC)
IV/nonce фиксированный или переиспользуется → 🔴
Encrypter(Key.fromUtf8('sabitanahtar')) → 🔴 встроенный в код ключ шифрования (тот же класс, что A1)
генерация token/ключа/OTP через Random() → 🟠 (для крипто-задач обязателен Random.secure())
base64/XOR/Caesar как «шифрование»       → 🔴 (encoding ≠ encryption)
самописный кастомный алгоритм шифрования  → 🔴 (никогда custom crypto)
хеш пароля на клиенте/в БД в открытом виде или без соли → 🔴 (если используется Firebase/Supabase Auth, они это уже делают; в custom auth — bcrypt/scrypt/Argon2)
```

**Правила:**
- Ключ шифрования никогда не находится в коде/asset → Keystore/Keychain (`flutter_secure_storage`) или выводится с сервера.
- Если нужен хеш — SHA-256+; для пароля — bcrypt/scrypt/Argon2 (даже одного SHA недостаточно).
- Ключ `HiveAesCipher` должен генерироваться через `Hive.generateSecureKey()` и храниться в secure storage (связано с A3).
- Примечание: Если в проекте вообще нет ручного крипто, этот пункт «НЕ ПРИМЕНИМО» — SDK Firebase/Supabase корректно управляют своим TLS/токен-крипто; **не писать собственный крипто-код — уже лучшая практика**, отмечай это не как находку, а как положительную заметку.

## A12 — 🟡 Утечка данных через экран / буфер обмена / клавиатуру / память (OWASP M6: Inadequate Privacy Controls)

В приложениях с чувствительными экранами (оплата, идентификация, личные сообщения, баланс):

**Сканирование:**
- **Блокировка скриншотов/записи экрана:** Есть ли на Android на чувствительных экранах `FLAG_SECURE` (или пакет вроде `screen_protector`/`no_screenshot`)? Если нет, вредоносный/вспомогательный сервис доступности, записывающий экран, увидит всё → в чувствительном приложении 🟡/🟠.
- **Снимок в переключателе приложений (app switcher):** iOS при переходе в фон делает фото экрана и показывает в app switcher; Android «Recents» делает то же самое. Маскируется ли чувствительный экран при переходе в фон (overlay/blur при `AppLifecycleState.paused/inactive`)? → если нет, 🟡.
- **Буфер обмена (clipboard):** Копируются ли через `Clipboard.setData(...)` токен/пароль/IBAN/персональные данные? На Android другие приложения (до 13) могут читать буфер обмена; если копирование необходимо, стоит рассмотреть очистку по таймеру → 🟡.
- **Кеш клавиатуры / autofill:** Установлены ли на чувствительных полях (`TextField`) `obscureText: true` + `enableSuggestions: false` + `autocorrect: false`? Сторонние клавиатуры запоминают и кешируют вводимое; корректен ли `autofillHints: [AutofillHints.password]` в поле пароля? → если чего-то не хватает, 🟡.
- **Превью уведомления:** Есть ли уведомление, показывающее чувствительный контент на экране блокировки (см. C6 FCM)?
- **Чувствительный state, остающийся в памяти после logout (🟡):** При вызове `signOut()` выполняется только UI-редирект, или чувствительный state в памяти (напр. `UserBloc`, `WalletState`, `ProfileNotifier`, объекты `riverpod`/`provider`/`bloc`, хранящие токен/PII) **явно сбрасывается / вызывается `close()` / `dispose()`?** Если не сбрасывается, то **на общем устройстве** (смена аккаунта, пока приложение живо как процесс) следующий пользователь может увидеть данные предыдущего на экране/в state → 🟡. *Реалистичное примечание:* В Dart гарантированно стереть память невозможно (GC + неизменяемость строк); поэтому это не «защита от memory dump», а пункт про **гигиену сессии** — главная угроза не атакующий с root, снимающий дамп RAM, а следующий человек, использующий то же устройство. Критично: при logout очистить дерево state + `clearPersistence()` (A3) + удалить данные сессии из secure storage.

## A13 — 🟠 Уязвимости MethodChannel / нативного моста (Kotlin/Swift) — Native Injection и Path Traversal

Этот пункт применяется, если в проекте есть собственные `MethodChannel` / `EventChannel` / `BasicMessageChannel` или написанный вручную нативный код (Kotlin/Java, Swift/Obj-C). В проекте, использующем только готовые SDK Firebase/Supabase и не имеющем ни одного собственного платформенного канала → **НЕ ПРИМЕНИМО.**

**Сканирование:**
- На стороне Dart найди вызовы `MethodChannel('...')`, `invokeMethod(...)`, `EventChannel(...)`; сопоставь их аналоги на стороне Android (`android/app/src/main/kotlin|java/.../*.kt|*.java`, `MethodChannel(...).setMethodCallHandler` внутри `configureFlutterEngine`) и iOS (`ios/Runner/*.swift`, `FlutterMethodChannel(...)`).
- Аргументы, приходящие от клиента (`call.arguments`, `call.argument("x")`), **без валидации** на нативной стороне утекают в:
  - SQL/SQLite-запрос (конкатенация строк → **SQL injection**),
  - путь к файлу (`File(path)`, `FileInputStream`, `NSFileManager` → **path traversal**; выход за пределы sandbox приложения через `../../`),
  - параметр `Intent` (Android) / `startActivity` / `PendingIntent` (→ intent injection, запуск неавторизованного компонента),
  - `Runtime.exec` / `ProcessBuilder` (→ инъекция команд),
  - `WebView.loadUrl` / `evaluateJavascript` (→ XSS / злоупотребление JS-мостом).

**Красный флаг:**
```
// Android (Kotlin)
db.rawQuery("SELECT * FROM t WHERE id = " + call.argument<String>("id"), null)  → SQL injection
File(context.filesDir, call.argument<String>("name")!!)   // нет проверки ../ / canonical → path traversal
Runtime.getRuntime().exec(call.argument<String>("cmd"))   → инъекция команд
// iOS (Swift)
let p = docsDir.appendingPathComponent(args["name"] as! String)  // нет проверки prefix/canonical
```

**Правило (в русле основной философии отчёта):** Нативный код тоже **часть клиента** — выполненная там валидация тоже обходится через обратную разработку/хук. То есть нужны сразу два слоя:
1. **Правильное написание** против injection/path-traversal на нативной стороне всё равно обязательно (параметризованный запрос, path canonicalization + проверка `startsWith` для корня sandbox, явный `Intent`, allow-list ввода) — это гигиена безопасности внутри клиента.
2. **Но решение о правах/владении/оплате здесь не принимается;** это решение проверяется на backend. Не считай MethodChannel «безопасным внутренним API» — атакующий может вызвать нативную функцию и напрямую (через хук). Утверждению вида «этот пользователь авторизован», приходящему из канала, доверять нельзя.

# РАЗДЕЛ B — ПОТОКИ АУТЕНТИФИКАЦИИ (Вход / Регистрация / Сессия)

Этот раздел — OWASP M3 (Insecure Authentication/Authorization). Независимо от Firebase Auth / Supabase Auth / custom, **сам поток входа** — наиболее эксплуатируемая поверхность. Нижеследующее применяется в зависимости от того, какой провайдер есть в проекте.

## B1 — 🟠 Защита от перечисления email (Firebase Auth)

В Firebase Auth/Google Identity Platform `createAuthUri` / `fetchSignInMethodsForEmail` раскрывают, зарегистрирован ли email → атакующий отсеивает утёкший список email по признаку «у кого есть аккаунт на этой платформе» → таргетированный фишинг + credential stuffing. Google после **15 сентября 2023** включил **email enumeration protection** по умолчанию в проектах, созданных после этой даты; `fetchSignInMethodsForEmail` устарел (deprecated).

**Сканирование:**
- Есть ли в коде использование `fetchSignInMethodsForEmail(...)`? Если есть → поток auth зависит от этого deprecated/отключённого поведения, 🟠 (сломается + риск перечисления). Если паттерн «сначала введи email, реши, вход это или регистрация» опирается на этот API, его нужно перепроектировать.
- **РУЧНАЯ ПРОВЕРКА:** Firebase Console → Authentication → Settings → включён ли «Email enumeration protection»? (В старых проектах нужно включать вручную.)

## B2 — 🟠 Тихое создание аккаунта при социальном входе и путаница провайдеров (Google/Apple)

Критическое поведение: при входе через Google/Apple/Facebook с помощью `signInWithCredential` / `signInWithPopup`, если совпадающего аккаунта нет, Firebase **тихо создаёт новый аккаунт**. `isNewUser` (AdditionalUserInfo) возвращается только при первом создании. В примерах вроде Bumble: атакующий может сначала открыть аккаунт с **ещё не зарегистрированным** email жертвы, а затем захватить его.

**Сканирование (если в проекте есть `google_sign_in` / `sign_in_with_apple` / Firebase social):**
- **Обрабатывается ли** ошибка `account-exists-with-different-credential` в потоке входа Google/Apple? Если не обрабатывается, аккаунты двух разных провайдеров с одним email вносят путаницу в авторизацию → 🟠. (Должны быть настройка Firebase «One account per email address» + поток credential linking.)
- Вход через Apple: Apple может выдать **relay/приватный email** и возвращает **имя/email только в первый раз**. Если код при последующих входах снова ожидает эту информацию, он сломается; её нужно сохранить при первом разе.
- **Обязательность Apple Sign-In:** Если приложение предлагает другой социальный вход (Google/Facebook), то согласно App Store Guideline 4.8 / 5.1.1 **также должен предлагаться Вход через Apple** (иначе отклонение в магазине). Если в проекте есть Google, но нет Apple → 🟡 (риск магазина).
- Новые версии `google_sign_in` имеют API, отличный от старых версий (`GoogleSignIn.instance`, `authenticate()`, `authorizationClient`). Если миграция со старого паттерна `signIn()` не выполнена — ошибки компиляции/выполнения + поломка потока — проверь используемую версию и совместимость API.

## B3 — 🟠 Управление сессией / токенами

**Сканирование:**
- **Отзыв токена при logout:** Если при выходе очищается только локальный state, но `signOut()` не вызывается / refresh token не отзывается, то с токеном, оставшимся на устройстве, сессию можно продолжить → 🟠. В Firebase — `FirebaseAuth.instance.signOut()`, в чувствительном сценарии дополнительно серверный отзыв токена. (Вместе с A12 проверь, что при logout очищается и чувствительный state в памяти.)
- **Персистентность refresh token:** Refresh token генерирует бесконечные idToken, пока не отозван. В чувствительном приложении при выходе/смене пароля он должен отзываться.
- **Проверка email:** Привязаны ли чувствительные операции к условию `emailVerified == true`? Если нет, права получают с неподтверждённым email.
- **MFA:** Есть ли многофакторная аутентификация для аккаунтов высокой ценности (оплата, admin)? (Требуется апгрейд Firebase → Identity Platform.)
- **Анонимная auth:** Должна использоваться только для временного state, не заменять постоянный аккаунт; анонимные пользователи тоже должны ограничиваться security rules («anyone can create anonymous account»).
- **Brute force / rate limit:** Есть ли ограничение на попытки входа? (Firebase частично автоматически; в custom auth нужно вручную.)
- **Политика паролей:** Минимальная длина/сложность, проверка на утёкшие пароли (password policy Identity Platform) включена?

## B4 — 🟠 Авторизация ≠ Аутентификация (Broken Access Control)

В OWASP Web Top 10 2021/2025 **#1 Broken Access Control**. В мобильных приложениях главная брешь тоже здесь.
- Путаются ли «вошёл» (authenticated) и «может выполнить эту операцию» (authorized)? Проверка роли/владения выполняется **на сервере** (rules/policy/function) или только на клиенте через скрытие UI? Скрытие на клиенте — не безопасность.
- Route guard (go_router `redirect` / auto_route guard) защищает только UI; **доступ к данным** должен защищаться **правилом backend**.
- IDOR: Можно ли вытянуть документ/запись с `id` другого пользователя? (Напр. если `/users/{чужойId}` можно запросить с клиента, а правило не проверяет владение → 🔴.)

# РАЗДЕЛ C — BACKEND (FIREBASE + SUPABASE)

Бо́льшая часть мобильных нарушений происходит не на клиенте, а в **неправильно сконфигурированных правилах backend.** В 2024 году одно исследование обнаружило ~125 миллионов записей (из них 19 миллионов паролей в открытом тексте), открытых на 916 сайтах Firebase. Со стороны Supabase бо́льшая часть нарушений вызвана оставленным отключённым RLS.

> Примечание: Применяется тот подраздел, чей сервис есть в проекте; отсутствующий помечается как **«НЕ ПРИМЕНИМО»**.

## C1 — 🔴 Правила безопасности Firebase Firestore / Realtime DB / Storage

**Claude Code должен прочитать следующие файлы:** `firestore.rules`, `database.rules.json`, `storage.rules`, `firebase.json` (какой файл rules деплоится, показывает `firebase.json`).

**Самый опасный паттерн (test mode утёк в продакшен):**
```
allow read, write: if true;         → 🔴 кто угодно читает/пишет что угодно
match /{document=**} { allow read, write: if true; }  → 🔴 вся БД открыта
".read": true, ".write": true        → 🔴 (Realtime DB)
```
Правила Firestore «test mode» **закрываются через 30 дней**, но срок мог быть продлён и перенесён в продакшен; кроме того, дата-зависимое временное правило вроде `allow read, write: if request.time < timestamp.date(2025,...)` в продакшене → 🔴.

**Вторая по частоте ошибка — «любой, кто вошёл, имеет доступ ко всему»:**
```
allow read, write: if request.auth != null;   → 🟠
```
`request.auth != null` говорит лишь «вошёл», но не ограничивает, **к каким данным** он может получить доступ. Другой пользователь тоже вошёл.

**Правильный паттерн (владение + валидация данных):**
```
match /users/{userId} {
  allow read, update, delete: if request.auth != null && request.auth.uid == userId;
  allow create: if request.auth != null && request.auth.uid == userId;
}
match /posts/{postId} {
  allow read: if true;                                        // объявления читаются публично
  allow create: if request.auth != null
                && request.resource.data.authorId == request.auth.uid
                && request.resource.data.title.size() < 200;  // серверная валидация
  allow update, delete: if request.auth != null
                && resource.data.authorId == request.auth.uid; // только владелец
}
```

**Claude Code должен для каждой коллекции спросить:**
1. Есть ли `allow ... if true` или дата-зависимое временное правило? → 🔴
2. Только `auth != null`, или есть проверка владения/совпадения UID? → если нет, 🟠
3. Ограничены ли `update`/`delete` владельцем? (Возможность изменить чужое объявление/документ = целостность данных + путь к захвату аккаунта)
4. Есть ли в `create`/`update` **валидация полей** (`request.resource.data...` тип/длина, неизменяемые поля `resource.data.x == request.resource.data.x`)? Может ли пользователь сам записать `role: admin` или `isPremium: true` → 🔴 повышение привилегий.
5. **Правила `get` и `list` (query) различны.** Правило, разрешающее чтение одного документа по ID, не даёт автоматически разрешения **запрашивать (query/list)** коллекцию; в местах, где выполняется query, правило должно выполняться для всех документов, которые может вернуть запрос. Ограничен ли доступ `list`/query владением/фильтром, или это «любой с auth != null запрашивает всю коллекцию»? → неограниченный query 🟠/🔴.
6. **Правила Storage отдельны** (`storage.rules`) — Firestore может быть безопасен, а Storage открыт. Есть ли ограничение размера файла (`request.resource.size < 5*1024*1024`) и типа (`request.resource.contentType.matches('image/.*')`)? Если нет — злоупотребление квотой/вредоносный файл.
7. Есть ли **ограничение частоты** записи / защита от abuse? (Атакующий может писать мусорные данные и взорвать счёт/лимит.)

**Дополнительный слой — App Check (🟠, очень важно):** Включён ли **App Check** для Firestore/Storage/RealtimeDB/Functions/Auth? Он проверяет, что запрос пришёл из твоего реального, неизменённого приложения (Android: Play Integrity, iOS: App Attest); отсеивает поддельные запросы, отправляемые скриптами. Проверь, есть ли в коде `FirebaseAppCheck.instance.activate(...)`. Если нет, то даже при правильных security rules `anon`/автоматизированные запросы свободны → 🟠.

## C2 — 🟡 Безопасность Firebase Remote Config

Если используется `firebase_remote_config` (напр. обязательная версия / feature flag):
- **Remote Config НЕ является хранилищем секретов.** Значения загружаются на клиент и читаемы; помещать API-ключ/секрет/URL → 🔴.
- **Force-update / feature gate обходится на клиенте.** Логика «блокируй тех, кто ниже такой-то версии» существует ради UX; **решение о безопасности** (премиум ли, имеет ли доступ) не должно на неё опираться — должно проверяться на сервере.
- Если условия Remote Config таргетируют по PII (полный email/телефон) — заметка о приватности.

## C3 — 🔴 Supabase Row Level Security (RLS)

В Supabase **золотое правило: на каждой таблице схемы `public` RLS должен быть включён, без исключений.** Ключ `anon` публичен; если RLS отключён, с этим ключом любой читает/пишет/удаляет всю таблицу.

**Claude Code должен просканировать следующее:**
- `supabase/migrations/*.sql`, `schema.sql`, `.sql`-файлы
- Есть `CREATE TABLE`, но за ним **нет** `ENABLE ROW LEVEL SECURITY` → 🔴
- Таблицы, созданные через SQL-редактор/миграцию, **не включают RLS автоматически** (в отличие от дашборда). Если в миграции этого явно нет → 🔴

**Проверочный запрос (РУЧНОЙ, пусть запустит пользователь):**
```sql
SELECT tablename FROM pg_tables
WHERE schemaname = 'public' AND NOT rowsecurity;
```
Каждая возвращённая таблица = незащищена = 🔴.

**Паттерны, где RLS включён, но политика ошибочна:**
```sql
-- 🔴 политика, «которая выглядит как разрешающая, но открывает всё»
CREATE POLICY "read" ON profiles FOR SELECT USING (true);

-- ✅ правильно: только своя строка
CREATE POLICY "read own" ON profiles FOR SELECT USING (auth.uid() = user_id);
```

**Другие критические проверки:**
1. **Ключ `service_role` на клиенте?** (`--dart-define`, в коде, `NEXT_PUBLIC_*`) → 🔴, полностью обходит RLS.
2. **Политика, опирающаяся на claim `user_metadata`** → 🔴. Это поле пользователь может изменить сам; если оно используется в проверке прав — повышение привилегий. Нужно использовать `app_metadata`.
3. **Если политика UPDATE не содержит `WITH CHECK`** → пользователь может переместить строку другому пользователю/tenant. В multi-tenant `tenant_id`/`org_id` должны защищаться через `WITH CHECK`.
4. **Можно ли вызвать RPC / Edge Function с `anon`?** Если функции admin/лицензий/оплаты можно вызвать публично → 🔴.
5. **Storage-бакеты:** Бакет, помеченный `public` = любой читает все файлы. Нужно проверить политики бакетов и дизайн пути к файлу (tenant/uid в пути).
6. **Утечка `service_role` в Edge Function:** логирование всего `env` на старте, возврат строки подключения/секрета в сообщении об ошибке.

## C4 — 🟠 Риск смешанного использования Firebase + Supabase (или двойного backend)

Если в одном приложении есть **и Firebase Auth, и Supabase DB** (или наоборот): как политики RLS Supabase проверяют идентичность Firebase? Если мост идентичности между двумя системами (проверка JWT, custom claims) неправильный, то тот, кто authenticated на одной стороне, может получить неавторизованный доступ на другой. **Идентичность должна управляться из единого источника (single source of truth);** проверка подписи JWT (audience/issuer) должна быть полной.

## C5 — 🟠 Безопасность Cloud Functions / Edge Functions

Если в проекте есть `functions/` или `supabase/functions/`:
- Проверяют ли функции идентичность/права, или это «вызвали — работает»? В не-callable HTTP-функциях проверку auth нужно делать вручную.
- Настроены ли валидация ввода, rate limit, `maxInstances` (против DOS, взрывающего счёт)?
- **Секреты в Vault/Secret Manager или в голом env?** Даже переменная окружения (env var) может утечь в логи функции, трассировки ошибок или в строку вроде `console.log(process.env)`. Настоящие секреты нужно хранить в **Google Cloud Secret Manager / Firebase secrets (`defineSecret`) / Supabase Vault** и передавать функции **по ссылке**; не в исходном коде и не в голом `.env`. Код, логирующий весь `env` на старте → 🔴.
- Не слишком ли ослаблена настройка CORS (`*`)?
- Не утекают ли в ответах об ошибках stack trace / информация о подключении?

## C6 — 🟠 Безопасность Push-уведомлений (FCM)

Если в проекте есть `firebase_messaging`:
- **Есть ли чувствительные данные в payload?** Содержимое уведомления видно на экране блокировки, проходит через серверы Google/Apple и может логироваться на устройстве. Содержимое OTP/токена/баланса/личного сообщения **не помещается в payload**; уведомление говорит «у тебя новое сообщение», а содержимое подтягивается по безопасному каналу при открытии приложения (паттерн data-only push + fetch). → При нарушении 🟠.
- **Безопасность topic:** На FCM topic **может подписаться кто угодно** — если имя `subscribeToTopic('...')` на клиенте предсказуемо (напр. `user_12345`, `admin`, `premium`), атакующий тоже подписывается и получает каждое уведомление, идущее в этот topic. **Данные, предназначенные пользователю/группе, должны отправляться не через topic, а по токену (с сервера на одно устройство).** → Если персональный контент отправляется через topic, 🟠.
- **Место хранения FCM-токена:** Токен обычно хранится в Firestore как `users/{uid}/fcmToken`. Если это поле **может читать/писать кто-то другой** (см. правила C1), атакующий может изменить токен и перенаправить уведомления жертвы на своё устройство → 🟠.
- **Серверная отправка:** Используется Legacy Server Key (постоянный, при утечке кто угодно шлёт push) или **HTTP v1 API + сервисный аккаунт (короткоживущий OAuth-токен)**? Если legacy key встроен в код/клиент → 🔴. (Google закрыл legacy FCM API; если он всё ещё используется, значит он уже сломан.)
- Если при клике по уведомлению обрабатывается deep link — действуют проверки A10/D2.

## C7 — 🟠 Проверка In-App Purchase / Premium + мониторинг счёта и злоупотреблений

**IAP / premium (если в проекте есть `in_app_purchase`, RevenueCat, `purchases_flutter` или поле вроде `isPremium`/`isPro`):**
- **Где определяется** статус premium? Если клиент может сказать «я купил» и записать `isPremium: true` в Firestore/Supabase → 🔴 (то же повышение привилегий, что в пункте 4 C1/C3). Правильно: **чек (receipt/purchase token) проверяется на сервере** (Cloud Function → Google Play Developer API / App Store Server API или webhook RevenueCat), и поле premium пишет **только сервер**; security rules запрещают клиенту писать это поле.
- Если информация о цене/праве продукта зафиксирована на клиенте, ей можно манипулировать; предоставление права всегда должно зависеть от серверной проверки.

**Мониторинг счёта и злоупотреблений (РУЧНАЯ ПРОВЕРКА):**
- Определено ли **бюджетное оповещение** (budget alert) GCP? Есть ли **сигнал об использовании** для Firestore/Functions/Storage? Атака (злоупотребление квотой, мусорная запись, функция в бесконечном цикле) чаще всего сначала проявляется в счёте; без сигнала её неделями не замечают.
- Стоит рассмотреть лимит `maxInstances` в Cloud Functions (C5) + сигнал об аномальном чтении/записи в Firestore.
- Официальный чек-лист безопасности Firebase рекомендует связаться с поддержкой при подозрении на DOS — отметь это как шаг реагирования на инцидент.

## C8 — 🟠/🔴 Доверие «заявленным» клиентом данным (Манипуляция геолокацией / счётом / временем / счётчиком)

Применяется в каждом потоке с геолокацией, геймификацией, лояльностью/баллами, шагомером, «check-in», расстоянием/наградой. Основная ошибка: **backend слепо принимает отправленные клиентом значения — lat/lng, счёт, timestamp, число шагов, расстояние.** Всё это генерируется на клиенте → **всё можно подделать** (Fake GPS / mock location, редактирование пакета для накрутки счёта, изменение часов устройства).

**Сканирование / вопросы:**
- Отправляет ли клиент на backend сырую геолокацию/счёт/время, и backend записывает это напрямую (в Firestore/Supabase `score`, `distance`, `location`)? → если проверки нет, 🟠; если по этому значению определяется премиум/награда/рейтинг, 🔴.
- Есть ли на сервере **проверка скорости / скачка (velocity / jump check)**? Физически ли возможны расстояние/время между двумя последовательными геолокациями (напр. 400 км за 2 минуты = невозможно)? Отклоняется ли внезапный скачок?
- Пытаются ли предотвратить Fake GPS / location mocking **только клиентским плагином** (флаг `isMockLocation`)? Этот клиентский сигнал **сам по себе недостаточен** (обходится) — основой должна быть серверная проверка согласованности (скорость, географические границы, серверный timestamp).
- Timestamp: используется ли в критических потоках **серверное время** (`FieldValue.serverTimestamp()` / DB `now()`), или время, отправленное клиентом? Доверие клиентскому времени → манипуляция рейтингом/наградой.
- Счёт/прогресс: если итоговый счёт вычисляется и отправляется на клиенте, им манипулируют; если критично, нужна серверная проверка / пересчёт.

**Правило:** Это из той же семьи, что A8 (валидация ввода) и B4 (авторизация): **ни один «факт», заявленный клиентом, не принимается за истину без проверки.** Backend должен проверять заявленные данные на согласованность своими независимыми сигналами (серверное время, предыдущее состояние, физическая логика).

# РАЗДЕЛ D — СПЕЦИФИЧНОЕ ДЛЯ ANDROID

Для каждого проекта, где есть каталог `android/`. Файлы: `android/app/src/main/AndroidManifest.xml`, `android/app/build.gradle(.kts)`, `android/app/proguard-rules.pro`, `res/xml/network_security_config.xml`.

## D1 — 🟠 Конфигурация Manifest
- `android:debuggable="true"` в release → 🔴 (обычно приходит из build type, если задано вручную — проверь).
- `android:allowBackup="true"` (по умолчанию) → данные приложения можно вытянуть через ADB backup. Если есть чувствительные данные — `false` либо исключить через `fullBackupContent` → 🟡/🟠. **Для Android 12+ (API 31) нужно дополнительно определить `android:dataExtractionRules`** — `fullBackupContent` сам по себе не покрывает сценарий переноса между устройствами (device transfer); файлы токена/сессии должны быть `exclude` в обоих наборах правил.
- **Tapjacking:** На экранах с чувствительными кнопками подтверждения (оплата, разрешение, удаление) касания можно обмануть через overlay. Есть ли на критических нативных view `filterTouchesWhenObscured` / на стороне Flutter — второй шаг (PIN/биометрия) при чувствительных подтверждениях? → если в чувствительном приложении нет, 🟡.
- **Task hijacking (StrandHogg):** Задан ли `android:taskAffinity` как пустая строка (`""`) и подходит ли `launchMode`? При taskAffinity по умолчанию вредоносное приложение может усесться поверх твоей задачи и показать поддельный экран входа → 🟡.
- `android:usesCleartextTraffic="true"` → 🟠 (см. A4).
- `activity`/`service`/`receiver`/`provider` с `android:exported="true"`: действительно ли они должны быть открыты вовне? Ненужный exported-компонент = запуск из другого приложения/утечка данных. Компоненты с `intent-filter` автоматически exported; пересмотри каждый из них.
- Собственный `ContentProvider` `exported` и без разрешения → утечка данных.

## D2 — 🟠 Проверка Deep/App Link
- Корректны ли `android:autoVerify="true"` в activity с `intent-filter` и `.well-known/assetlinks.json` на нужном хосте? Если нет, другое приложение может украсть ссылку (App Link hijack).
- Запускается ли чувствительная операция без проверки параметров приходящей ссылки (см. A10, B).

## D3 — 🟡 Сборка / уменьшение кода
- Включены ли `minifyEnabled true` + `shrinkResources true` в release?
- Защищают ли правила ProGuard/R8 чувствительные классы, не отменяют ли обфускацию ненужным `-keep`?
- Подпись: находится ли пароль keystore в открытом тексте в `build.gradle` внутри `signingConfigs` (→ 🔴), или приходит из `key.properties`/env? Находятся ли `key.properties` и `*.keystore`/`*.jks` **в `.gitignore`**?

## D4 — 🟡 Play Integrity
- Интегрирован ли в чувствительных приложениях Play Integrity API / App Check (Play Integrity provider)? Даёт ли детекция эмулятора/root/tamper сигнал на backend?

# РАЗДЕЛ E — СПЕЦИФИЧНОЕ ДЛЯ iOS

Для каждого проекта, где есть каталог `ios/`. Файлы: `ios/Runner/Info.plist`, `ios/Runner/Runner.entitlements`, `ios/Podfile`.

## E1 — 🟠 App Transport Security (ATS)
- `NSAppTransportSecurity` → `NSAllowsArbitraryLoads = true` — глобальный обход ATS → 🟠. Если нужно исключение, оно должно быть по домену через `NSExceptionDomains`, а не глобальным.

## E2 — 🟠 URL-схемы и Universal Links
- Валидируются ли потоки, открываемые через `CFBundleURLTypes` (custom scheme)? Custom scheme открыт для hijack; для чувствительного потока следует предпочесть **Universal Links**.
- Корректен ли `Runner.entitlements` → `com.apple.developer.associated-domains` (`applinks:`)? Находится ли файл `apple-app-site-association` на хосте?

## E3 — 🟡 Keychain и приватность
- Хранятся ли чувствительные данные в Keychain (secure_storage использует его), или в открытом виде в `NSUserDefaults`/plist? (см. A3.)
- Уровень доступности Keychain: `kSecAttrAccessibleWhenUnlocked` (или строже, напр. `...WhenPasscodeSetThisDeviceOnly`), или ослабленный вроде `...Always`/`AfterFirstUnlock`? Ослабленный уровень = на устройстве с jailbreak легче прочитать через `objection > ios keychain dump`.
- **Биометрия — не единственная защита:** iOS-биометрическую проверку (LocalAuthentication) можно обойти в рантайме через objection/Frida. Биометрическое «подтверждение» должно *держать* критические данные заблокированными (данные должны быть зашифрованы флагами `SecAccessControl` так, чтобы их нельзя было расшифровать без биометрии — напр. пакет `biometric_storage` использует `SecAccessControlWithFlags`), одной логики «if (authenticated) показать» недостаточно.
- Полны ли строки описания разрешений в `Info.plist` (`NS...UsageDescription`) (падение + отклонение в магазине).
- Актуальны ли **Privacy Manifest** (`PrivacyInfo.xcprivacy`) и декларации «required reason API»? (Apple требует с 2024 года; если отсутствуют — отклонение в магазине → 🟡.)

## E4 — 🟡 Pod-зависимости
- Закоммичен ли `Podfile.lock`? Закреплены ли поды, есть ли поды с известными CVE?
